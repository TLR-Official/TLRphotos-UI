
# Trae 全局强制规则 (Global Mandatory Rules)

> ⚠️ 本文件为最高优先级指令。任何任务、任何对话轮次均必须遵守，不得因上下文切换、任务变更或用户未显式提醒而忽略。

## 1. 核心架构基准 (Architecture Baseline)

以下约束在任何任务中均不可违反，即使用户明确要求也需先警告风险：

- 🚫 **彻底废弃 NocoBase**：禁止引入、修复、配置或讨论任何低代码/无代码平台、Headless CMS 及 GUI 数据库管理工具。
- 🚫 **禁止服务器前端开发**：禁止在生产/测试服务器安装 Node.js 前端工具链、运行 Vite dev server 或开放前端开发端口。所有前端开发与联调必须在本地 Monorepo 工作区完成。
- 🚫 **禁止猜测 API**：在编写或修改前端数据请求代码前，必须先读取 `backend/docs/api.md`。若文件不存在或接口未定义，必须先生成后端代码并更新文档，严禁凭训练数据捏造接口。
- 🚫 **禁止额外安全组件**：禁止引入 SafeLine / ModSecurity / 独立 WAF 容器。
- 🚫 **禁止原图本地存储**：原图必须走 Cloudflare R2 预签名 URL，禁止存于服务器磁盘。
- ✅ **纯代码后端**：后端必须使用纯代码实现（Node.js + Express/Fastify + better-sqlite3），所有模型、路由、权限以 `backend/` 目录代码为唯一事实来源。
- ✅ **UI 壳子保留**：前端仅保留现有 JSX/Tailwind 静态布局与纯展示组件，数据层必须与后端 `api.md` 契约严格对齐。
- ✅ **安全防护**：管理后台必须经 Cloudflare Zero Trust Access 保护；缩略图存为 WebP Blob 至服务器独立数据盘。
- ✅ **配置注入**：所有环境变量通过 CI/CD 或 `.env` 注入，禁止硬编码敏感信息。

## 2. Context 文件同步协议 (CRITICAL)

### 2.1 触发条件

**每一次**对代码库产生实际变更的操作完成后（新建/修改/删除文件、调整配置、执行命令、修复Bug、重构代码），**必须立即**执行 Context 更新。

- ✅ 触发：写入代码、修改配置、执行构建/测试命令、更新 api.md
- ❌ 不触发：纯问答、解释说明、未落地的方案讨论

### 2.2 更新格式

在 `.ai/context.md` 文件的 `## Changelog` 章节**顶部**追加条目，严格使用以下格式：

| YYYY-MM-DD HH:mm | [类型] 简要描述（≤30字） | 关联文件/模块 |

- **时间戳**：使用 Asia/Shanghai 时区，精确到分钟
- **类型标签**：仅限 `feat` / `fix` / `refactor` / `config` / `docs` / `chore`
- **描述**：动词开头，说明"做了什么"而非"为什么做"
- **关联**：列出受影响的核心文件或模块路径（含 api.md 更新）

### 2.3 执行保障

- 更新 Context 是变更操作的**组成部分**，不是附加任务
- 若单次操作涉及多处变更，合并为一条 Changelog 记录
- 禁止使用"稍后更新""记得提醒我"等延迟表述
- 若 Context 文件不存在，先创建标准模板再写入

## 3. Git 推送强制协议 (CRITICAL)

### 3.1 触发条件

**每一次**完成用户明确要求的变更 **且** Context Changelog 更新完毕后，**必须立即**执行 Git 提交与推送。

- ✅ 触发：用户指令已落地 + Changelog 已写入 + 本地联调通过
- ❌ 不触发：纯讨论/方案确认/Changelog 未更新/代码检查失败/未完成 api.md 同步

### 3.2 执行流程（严格按序）

1. **检查**：运行项目配置的 lint / type-check / build 命令，**任一失败则中止推送**并报告错误
2. **暂存**：仅 `git add` 本次变更相关文件，**禁止 `git add .` 或 `git add -A`**
3. **提交**：Commit Message 格式为 `[类型] 简要描述 | YYYY-MM-DD HH:mm`，与 Changelog 条目对应
4. **推送**：执行 `git push origin <当前分支>`，**禁止 `--force` / `--force-with-lease`**
5. **确认**：向用户反馈推送结果（成功/失败及原因）

### 3.3 安全红线

- 🚫 禁止推送含 `.env` / `*.key` / `*.db` / `node_modules/` / `dist/` 等敏感或生成文件
- 🚫 禁止在主分支（main/master）上直接推送未经用户确认的变更
- 🚫 禁止在推送前跳过步骤1的检查流程
- 🚫 禁止使用模糊 Commit Message（如 "update" / "fix" / "changes"）

### 3.4 异常处理

- 若推送因网络/权限失败：**重试1次**，仍失败则报告错误并保留本地提交，**不得丢弃变更**
- 若检查流程失败：**停止推送**，输出完整错误日志，等待用户指示
- 若用户要求"先别推"或"等下再推"：**立即暂停**，在 Changelog 中标注 `[push-deferred]`，待用户显式恢复后继续

## 4. 任务切换恢复协议

当检测到新任务与上一任务无关时：

1. **先读取** `.ai/context.md` 全文与 `backend/docs/api.md`
2. **确认**当前架构状态、Changelog 最新条目及 API 契约版本
3. **再开始**新任务的规划与执行
4. 禁止假设上一任务的上下文、NocoBase 旧记忆或未文档化的接口仍然有效

## 5. 规则优先级

本文件 > 用户临时指令 > AI 默认行为
若用户指令与本文件冲突，必须明确告知冲突点并获得显式确认后方可例外执行，且需在 Changelog 中标注 `[override]`。
